# [DRAFT v2] Version 4

## 1. Transport

The FCast protocol consists of having a receiver creating a TCP server listening socket, which senders use to establish a connection on another port for receiver control.
All communication is plaintext, designed for trusted local area networks.
Additional optional features may also require additional ports which are described in the table below.

### 1.1 Ports

Applications may allow using different port numbers, but the following designated ports are recommended for firewall predictability.

| Port  | Protocol | Side     | Purpose                |
|-------|----------|----------|------------------------|
| 46899 | TCP      | Receiver | Connection listener    |
| 46900 | TCP      | Sender   | WHEP signalling        |
| 46901 | TCP      | Sender   | Media proxy (HTTP)     |
| 46902 | UDP      | Both     | WebRTC media           |
| 5353  | UDP      | Both     | mDNS discovery         |

### 1.2 Packet Structure

#### Header

The header packet structure is defined as:

```rust
pub struct Header {
    pub size: u32,   // Little Endian
    pub opcode: u8,
}
```

For a packet with no body, only the header is sent. In this case, `size = 1`. When a body is present, it has `size - 1` bytes following the header.

The maximum body size is 64KB. Implementations should allocate a 64KB socket read buffer to accommodate the header and maximum body size.

### 1.3 Opcodes

The `opcode` field in the header identifies the message type. Each opcode has a fixed direction and an optional body structure.

| Opcode | Name        | Direction          | Body                                                |
|--------|-------------|--------------------|-----------------------------------------------------|
| 11     | Version     | Both               | `VersionMessage`                                    |
| 12     | Ping        | Both               | None                                                |
| 13     | Pong        | Both               | None                                                |
| 14     | Initial     | Both               | `InitialSenderMessage` / `InitialReceiverMessage`   |
| 20     | Play        | Sender -> Receiver | `PlayMessage`                                       |
| 21     | SetPlayback | Sender -> Receiver | `SetPlaybackMessage`                                |
| 22     | StateUpdate | Receiver -> Sender | `StateUpdateMessage`                                |
| 23     | Queue       | Both               | `QueueMessage`                                      |
| 24     | Error       | Both               | `ErrorMessage`                                      |

??? info "Opcodes not used in v4 sessions"
    Opcodes 0-10, 15-19 from previous versions are not used when both parties negotiate v4. They remain active when session falls back to older protocol versions.

### 1.4 Message Body

When a message has a body (`size > 1`), it is a JSON object encoded as UTF-8. The structure of the body is determined by the opcode. Message body definitions are described in [Messages](#4-messages).

The following serialization rules apply:

- Fields with null values may be omitted
- Parsers must ignore unknown fields to allow for forward compatibility
- Parsers must not crash on unknown enum variants. They should treat them as unknown/unsupported.


## 2. Connecting

Senders can discover and connect receivers on the local network via mDNS. QR code and/or manual IP entry connection methods should also be supported.

### 2.1 mDNS

Receivers supporting mDNS service advertisements should use the service type `_fcast._tcp.local.`.
Additionally the following TXT records are recommended to include in case senders wish to use the following metadata before connection:

| Key            | Value                        |
|----------------|------------------------------|
| `version`      | Protocol version (e.g. `4`)  |
| `displayName`  | Human-readable device name   |
| `appName`      | Application name             |
| `appVersion`   | Application version          |

### 2.2 QR Code

Receivers may display a QR code encoding a connection URL:

```
fcast://r/{base64-encoded-json}
```

The JSON structure with example values:

```json
{
  "name": "Device Name",
  "version": 4,
  "addresses": ["192.168.1.100", "fe80::1"],
  "services": [
    {"port": 46899, "type": 0}
  ]
}
```

Where `type` 0 indicates TCP. Parsers should ignore unknown fields in the QR code JSON to allow for future extensions.

## 3. Connection Lifecycle

Overview of the connection lifecycle from establishment to active session:

``` mermaid
sequenceDiagram
  participant S as Sender
  participant R as Receiver

  S->>R: TCP connect
  par Both send Version
    S->>R: Version {version: 4}
    R->>S: Version {version: 4}
  end
  Note over S,R: negotiated_version = min(sender.version, receiver.version)
  par Both send Initial
    S->>R: Initial (device info)
    R->>S: Initial (device info, capabilities)
  end
  R->>S: StateUpdate (full state snapshot)
  S->>R: Play, SetPlayback, Queue ...
  R->>S: StateUpdate, Error ...
```

### 3.1 Connection Establishment

After a TCP connection is established, the assumed protocol version is v2 until version negotiation takes place. Both parties **must** immediately send a `Version` message upon connection.
Both parties compute the negotiated version in the following manner:

```
negotiated_version = min(sender.version, receiver.version)
```

If the negotiated version is not supported by either party, it should either downgrade to a compatible feature set or disconnect.
If both parties support v4 or higher, an `Error` message with code `Incompatible` should be sent before disconnecting.

After version negotiation, both parties **must** send an `Initial` message with device information and capabilities if supported by the protocol version.
The receiver **must** then immediately send a `StateUpdate` with a full state snapshot to synchronize the sender with the current playback state.

Both parties should periodically send `Ping` messages to detect connection loss. The other party must respond with `Pong`. If no response is received within a reasonable timeout, the connection should be considered dead.

### 3.2 Disconnection and Errors

For graceful disconnection, a party should send an `Error` message with code `Closing` before closing the TCP connection. This allows the other party to display a reason to the user if desired.

If no `Error` message is received before the TCP connection closes, the disconnection is assumed to be unexpected (crash, network failure, etc.).

Error codes marked as fatal (`Closing`, `Incompatible`, `Timeout`, `Refused`) indicate the sending party will close the connection after sending. All other error codes are non-fatal and the connection remains open.

### 3.3 Multiple Senders

The protocol supports multiple senders connected to a single receiver simultaneously. Each sender has an independent TCP connection and session.
If the receiver does not allow multiple senders or reached its maximum sender limit, additional connections should be refused with an error code `Refused`.

## 4. Messages

### 4.1 `VersionMessage`

Sent by both parties immediately after TCP connection for version negotiation.

```rust
pub struct VersionMessage {
    pub version: u64,
}
```

### 4.2 `InitialMessage`

Sent by both parties after version negotiation to indicate device information and capabilities.

The sender sends device identification:

```rust
#[skip_serializing_none]
pub struct InitialSenderMessage {
    pub display_name: Option<String>,
    pub app_name: Option<String>,
    pub app_version: Option<String>,
}
```

The receiver sends device identification and capabilities:

```rust
#[skip_serializing_none]
pub struct InitialReceiverMessage {
    pub display_name: Option<String>,
    pub app_name: Option<String>,
    pub app_version: Option<String>,
    pub capabilities: Option<Capabilities>,
}
```

### 4.3 `PlayMessage`

Sent by the sender to start playback of a single media item or a queue.

```rust
#[skip_serializing_none]
pub enum PlayMessage {
    #[serde(rename = "SINGLE")]
    Single {
        item: MediaItem,
    },
    #[serde(rename = "QUEUE")]
    Queue {
        items: Vec<QueueItem>,
        /// Index of the item to begin playback from
        start_index: u32,
        /// Total number of items in the queue (None = items array is the complete queue)
        total_items: Option<u32>,
        /// Default HTTP headers applied to all items
        headers: Option<HashMap<String, String>>,
        /// Default volume for all items (0.0-1.0)
        volume: Option<f64>,
        /// Default playback speed for all items
        speed: Option<f64>,
        /// Default display duration in seconds
        show_duration: Option<u32>,
        /// Queue-level metadata
        metadata: Option<MetadataObject>,
        /// Repeat mode
        repeat: Option<RepeatMode>,
        /// Shuffle playback order
        shuffle: Option<bool>,
    },
}
```

A new `Play` message stops current playback and replaces the current media or queue entirely. To switch to a different item within the current queue, use `SetPlayback` with `item_index` instead.

The `headers`, `volume`, `speed`, and `show_duration` fields on the `Queue` variant serve as defaults for all items. If a `MediaItem` or `QueueItem` specifies its own value for any of these fields, the item-level value is used instead of the queue default.

When `total_items` is present, the receiver should begin playback immediately from `start_index` without waiting for all items. Remaining items arrive via subsequent `Queue INSERT` messages. When `total_items` is absent, the items array is the complete queue. If the sender disconnects before all items are received, the receiver treats the queue as complete at its current count.

??? tip "Implementation Note: Large Queues"

    For large queues that exceed practical single-message sizes, senders should deliver items incrementally and set the `total_items` expected in the full playlist.
    For example sending a queue with 200 items can be done as follows:

    ``` mermaid
    sequenceDiagram
      participant S as Sender
      participant R as Receiver

      S->>R: Play (QUEUE, items 0-49, total_items: 200)
      R->>S: StateUpdate (Playing, item_index: 0)
      S->>R: Queue INSERT (items 50-99, position: BACK)
      S->>R: Queue INSERT (items 100-149, position: BACK)
      S->>R: Queue INSERT (items 150-199, position: BACK)
      Note over R: Queue fully loaded (200/200)
    ```

    The same pattern is used by the receiver when synchronizing a newly connected sender with an existing large queue.

### 4.4 `SetPlaybackMessage`

Sent by the sender to modify playback state. All fields are optional, only include fields being changed. Multiple fields may be set in a single message for atomic changes.

Valid `state` values: `Playing` (resume), `Paused` (pause), `Idle` (stop playback). `Buffering` and `Ended` are receiver-only states and must not be sent by the sender.

```rust
#[skip_serializing_none]
pub struct SetPlaybackMessage {
    /// Change playback state
    pub state: Option<PlaybackState>,
    /// Seek to time in seconds
    pub time: Option<f64>,
    /// Change volume (0.0-1.0)
    pub volume: Option<f64>,
    /// Change playback speed
    pub speed: Option<f64>,
    /// Switch to a different queue item by index
    pub item_index: Option<u32>,
    /// Change video track
    pub video_track: Option<u32>,
    /// Change audio track
    pub audio_track: Option<u32>,
    /// Change subtitle track
    pub subtitle_track: Option<u32>,
    /// Load an external subtitle track during playback
    pub load_subtitle: Option<SubtitleSource>,
    /// Change repeat mode
    pub repeat: Option<RepeatMode>,
    /// Toggle shuffle
    pub shuffle: Option<bool>,
}
```

### 4.5 `StateUpdateMessage`

Sent by the receiver to all connected senders whenever playback state changes.

```rust
#[skip_serializing_none]
pub struct StateUpdateMessage {
    /// Unix timestamp in milliseconds when this update was generated
    pub generation_time: u64,
    /// Current playback state
    pub state: Option<PlaybackState>,
    /// Current playback position in seconds
    pub time: Option<f64>,
    /// Total duration in seconds
    pub duration: Option<f64>,
    /// Current volume (0.0-1.0)
    pub volume: Option<f64>,
    /// Current playback speed
    pub speed: Option<f64>,
    /// Current queue item index
    pub item_index: Option<u32>,
    /// Current repeat mode
    pub repeat: Option<RepeatMode>,
    /// Current shuffle state
    pub shuffle: Option<bool>,
    /// Available tracks (included when tracks change, e.g. on new media)
    pub tracks: Option<Tracks>,
    /// Current media/queue info (included only when the played content changes)
    pub play_data: Option<PlayMessage>,
}
```

Fields are optional. The receiver should only include fields that have changed or that are relevant to the current update. For example, a periodic time update need only include `generation_time`, `state`, and `time`.
The `play_data` field should only be included when the played content or queue changes.

When sent immediately after the `Initial` exchange to synchronize a newly connected sender, all fields should be populated as a full state snapshot including `play_data` if media is loaded.


### 4.6 `QueueMessage`

Sent by the sender to modify the queue. Relayed by the receiver to all other connected senders to keep queue state synchronized.

**Direction:**

- Sender -> Receiver: modify the queue
- Receiver -> Sender: notify that the queue was modified (by another sender or by the receiver itself)

```rust
#[skip_serializing_none]
pub enum QueueMessage {
    #[serde(rename = "INSERT")]
    Insert {
        items: Vec<QueueItem>,
        position: QueuePosition,
    },
    #[serde(rename = "REMOVE")]
    Remove {
        position: QueuePosition,
        /// Number of items to remove (default: 1)
        count: Option<u32>,
    },
}
```

### 4.7 `Ping / Pong`

Header-only messages with no body. Both parties should periodically send `Ping` to detect connection liveness. The other party must respond with `Pong`.
If no response is received within a reasonable timeout, the connection should be considered dead and the device should send an `ErrorMessage` indicating a timeout error and disconnect.

### 4.8 `ErrorMessage`

Bidirectional message for reporting errors and signalling disconnection.

```rust
#[skip_serializing_none]
pub struct ErrorMessage {
    pub code: ErrorCode,
    /// Human-readable error description
    pub message: Option<String>,
}
```
